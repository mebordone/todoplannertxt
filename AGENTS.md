# AGENTS.md

## Purpose

This document defines the mandatory quality standards for any code generated fully or partially by Large Language Models (LLMs) within this organization.

This policy applies to:
- Code generated via ChatGPT, Copilot, Cursor, or similar systems
- Code produced by autonomous coding agents
- Refactors or modifications suggested by LLMs
- Tests generated by LLMs

LLM-generated code is **not trusted by default** and must satisfy stricter validation rules.

---

# 1. Mandatory Human Review

1. All LLM-generated code MUST be reviewed by a human developer.
2. No automatic merge is allowed.
3. The human reviewer is fully responsible for:
   - Correctness
   - Security
   - Performance
   - Architectural consistency
4. If the author cannot explain the generated code, it MUST NOT be merged.

---

# 2. Test Coverage Requirements

LLM-generated code MUST include tests.

Minimum thresholds:

- Overall coverage: **≥ 90%**
- Branch coverage: **≥ 80%**
- Core business logic coverage: **≥ 95%**

Tests MUST:

- Include meaningful assertions
- Cover edge cases
- Cover error handling paths
- Cover invalid input scenarios
- Avoid trivial assertions made only to increase coverage

Code that reduces global coverage MUST be rejected.

No exceptions without explicit approval.

---

# 3. Cyclomatic Complexity Limits

LLMs frequently generate overly complex functions. Strict limits apply:

- Recommended per function: ≤ 7
- Maximum allowed per function: 10
- > 10 requires mandatory refactor before merge
- > 15 is strictly forbidden

Developers MUST refactor by:
- Splitting large functions
- Extracting helpers
- Removing nested conditionals
- Applying polymorphism or strategy patterns when applicable

---

# 4. Security Review Requirements

LLM-generated code MUST undergo explicit security validation.

Review MUST verify:

- No hardcoded credentials
- No embedded tokens
- No private keys
- Proper input validation
- Protection against injection attacks
- Proper authentication and authorization checks
- Safe error handling
- No sensitive data leakage in logs

If security implications are unclear, the code MUST be rejected.

---

# 5. Architectural Compliance

LLM-generated code MUST:

- Respect existing architectural layers
- Not introduce circular dependencies
- Not bypass domain boundaries
- Not introduce unnecessary frameworks or libraries
- Not duplicate existing logic

If the LLM output conflicts with architecture, it MUST be rewritten.

---

# 6. Dependency Policy

LLM-generated code MUST NOT:

- Add new external dependencies without justification
- Upgrade major versions automatically
- Introduce unmaintained libraries

All new dependencies require human approval.

---

# 7. Code Quality Requirements

Generated code MUST:

- Be readable and maintainable
- Use descriptive naming
- Avoid dead code
- Avoid unused imports
- Follow project formatting and linting rules
- Pass all CI checks

Generated code MUST NOT:

- Contain commented-out large code blocks
- Include placeholder TODO logic in production paths
- Contain speculative optimizations

---

# 8. Performance Responsibility

LLM-generated code MUST:

- Avoid obvious inefficiencies
- Avoid N+1 patterns
- Avoid unnecessary loops
- Avoid excessive memory allocations

Performance-sensitive code MUST be benchmarked if applicable.

---

# 9. Traceability

Pull Requests that include LLM-generated code SHOULD:

- Explicitly declare LLM assistance
- Include the prompt used (if non-trivial)
- Describe validation steps performed

Transparency is encouraged.

---

# 10. CI Enforcement

Continuous Integration MUST:

- Fail if coverage drops below thresholds
- Fail if complexity limits are exceeded
- Fail on linting errors
- Fail on security scanner findings

No bypass of CI protections is allowed.

---

# 11. Responsibility Model

LLMs are productivity tools.

They are not:
- Autonomous decision-makers
- Authorities on architecture
- Security auditors
- Performance experts

Final responsibility always belongs to human engineers.

---

# 12. Guiding Principle

Speed without quality increases long-term cost.

LLM-generated code is acceptable only when it:
- Improves productivity
- Maintains system integrity
- Does not introduce technical debt
- Meets or exceeds manual engineering standards
